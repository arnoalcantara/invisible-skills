# FICHA — Exemplo resolvido (worked example)

**Família:** F — Concretizar o abstrato · **ID:** `exemplo-resolvido`

**Função didática.** Mostrar a **solução completa de um problema modelo, passo a passo**, com o raciocínio à vista. É um dos achados mais sólidos da ciência da aprendizagem (o "worked-example effect"): para quem está aprendendo, estudar uma solução pronta carrega menos a memória do que tentar resolver do zero, e ensina mais. O trabalho na cabeça do aluno: ver *como se pensa* o problema, não só a resposta.

**Posição no arco.** Desenvolver, logo após apresentar um método/conceito. Idealmente seguido de um problema para o aluno resolver (a transição do worked example para a prática).

**Quando usar.** Há um procedimento, cálculo, análise ou raciocínio que o aluno precisa aprender a executar. Quando "ver alguém resolver" ensina mais que a regra abstrata.

**Quando NÃO usar.** Conteúdo puramente declarativo (sem "resolver" nada). Quando o aluno já domina o método (aí vá direto ao problema — [problema-para-resolver](problema-para-resolver.md)).

**Anatomia (slots).**
- *Título-asserção* (obrigatório) — o que o exemplo demonstra, ou o enunciado do problema.
- *Enunciado* (obrigatório) — o problema concreto, curto.
- *Passos da solução* (obrigatório) — cada passo com a ação **e o porquê** (a justificativa é o que ensina, não só a conta).
- *Resultado destacado* (obrigatório) — a resposta final, marcada.
- *Nota de armadilha* (opcional) — onde se erra neste tipo de problema.

**Regras de carga.** Um problema por slide (se a solução é longa, quebre em slides ou use build). Cada passo curto, com a justificativa em uma linha. Destaque visual no passo-chave. Não misture dois exemplos no mesmo slide.

**Build.** Fortemente recomendado: revele passo a passo, deixando o aluno acompanhar (ou prever o próximo passo antes de mostrá-lo). A revelação progressiva (lei 8) é central aqui — evita que o aluno pule para a resposta.

**Decisão visual (roteamento).** **Modo 0** — texto, fórmulas e marcações em HTML/CSS/SVG, com destaque de cor no passo crítico. Não roteia para gerador. Se o problema é visual (geometria, gráfico), o diagrama é desenhado em código.

**Conexão posicional.** *Pede antes:* o método/conceito já apresentado e o vocabulário pré-treinado. *Aponta para:* um problema análogo para o aluno resolver (transferência) ou uma verificação de entendimento.

**Erro comum.** Mostrar só a resposta, sem o raciocínio (perde o efeito worked-example). Pular passos "óbvios" que o iniciante não vê. Despejar a solução inteira de uma vez (sem build, o aluno lê a resposta e desliga). Dois exemplos competindo num slide.

**Por dial de fidelidade.** *Transcrição:* use o exemplo que o professor resolveu, com o método dele — inclusive os atalhos que ele usou. Não troque por uma solução "mais elegante". *Esboço:* você cria um problema modelo representativo, marcado para aprovação; confira a correção da solução.
